# AWS CloudTrail の要点

> 重要度: ○ ／ ANS-C01 では「誰が・いつ SG/ルート/VPC を変更したか」のネットワーク構成変更追跡と、VPC エンドポイント経由アクセスの監査が問われる。

## これは何
- AWS アカウント内の **API コール（誰が・いつ・何を・どこから）を記録する監査ログ**サービス。
- **コントロールプレーンの操作監査**を担う。トラフィックの中身や流量を見る [CloudWatch](cloudwatch.md) / VPC フローログとは守備範囲が異なる。

## 試験頻出ポイント
- **ネットワーク構成変更の追跡は「管理イベント」**。`AuthorizeSecurityGroupIngress` / `RevokeSecurityGroupIngress`、`CreateRoute` / `ReplaceRoute`、VPC/サブネット/ピアリング/TGW の作成・削除などはここ。**既定でオン**。
- **データイベント**は S3 オブジェクト・Lambda 実行などデータプレーン操作。高頻度・**既定オフ・追加課金**。ネットワーク構成追跡には基本不要。
- **ネットワークアクティビティイベント**で **VPC エンドポイント経由**の API コール（特に拒否 `VpceAccessDenied`）を監査。フローログ（IP メタデータのみ）では不可。**既定オフ・追加課金**。
  - 有効化は**高度なイベントセレクター**で `eventCategory = NetworkActivity` ＋ `eventSource`（例 `ec2.amazonaws.com`）。`errorCode = VpceAccessDenied` ＋ `vpcEndpointId` で特定エンドポイントの拒否のみ抽出可能。
- **イベント履歴は直近90日・無料**。それ以上の保持や横断分析は**証跡（S3）または CloudTrail Lake（SQL分析）**。
- **リアルタイム検知**: CloudTrail → **CloudWatch Logs → メトリクスフィルター → アラーム → SNS**。「想定外の SG 全開放（`0.0.0.0/0`）」等を即時通知。S3+Athena はバッチでリアルタイム性に欠ける。
- **組織証跡（Organization Trail）**: 管理アカウントで作成すると全メンバーアカウントを単一バケットに自動集約（[Organizations](organizations.md)）。個別作成・手動コピーは漏れの元。
- **マルチリージョン証跡**を有効にすると、新規リージョン追加時も自動的に記録対象になる。
- **Config との役割分担**: Config は「構成の状態（今どうか・どう変化したか）」、CloudTrail は「変更を行った API コール（誰が・いつ）」。両者は補完関係。

## CloudWatch との違い
| 観点 | CloudTrail | CloudWatch |
|---|---|---|
| 記録対象 | API 操作（コントロールプレーン） | メトリクス・ログ・トラフィック監視 |
| 答える問い | 誰がいつ何を変更したか | リソースの状態・性能はどうか |
| 主な用途 | 監査・セキュリティ調査・構成変更追跡 | 監視・アラーム・可視化 |
| 連携 | ログを CloudWatch Logs へ送り検知 | フィルター/アラームで CloudTrail を活用 |

## よく問われる上限・注意点（ひっかけ）
- **イベント履歴の90日閲覧は無料**。「課金される」は誤り。
- **管理イベントは1証跡コピー目のみ無料**、追加コピーは課金。「常に全コピー無料」は誤り。
- **データイベント／ネットワークアクティビティイベントは既定オフ＋追加課金**。SG 変更はデータイベントではなく管理イベントで記録される（「データ有効化が必要」は誤り）。
- 証跡数の上限はリージョンあたり **5**。
- マルチリージョン化自体に固定料金は発生しない。
- VPC エンドポイント越しのアクセス監査をフローログでやろうとするのは誤答（実行者・許可/拒否は分からない）。
- CloudTrail はパケットの中身・流量・レイテンシ・BGP 状態は扱わない（それぞれフローログ/ミラーリング/CloudWatch の領域）。
