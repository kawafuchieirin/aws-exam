# AWS RAM (Resource Access Manager) の要点

> 重要度: ○ ／ マルチアカウントNW設計の中核。「複数アカウントでTGW/サブネット/Resolverルールを共有」と来たら即RAM。

## これは何
- AWSアカウント間・Organizations/OU間・IAMロール/ユーザー間でリソースを安全に共有するサービス。
- 一度作ったリソースを共有すれば、コンシューマーはネイティブに存在するかのように利用できる。
- ネットワークではTGW・VPCサブネット・Resolverルール・Prefix List・IPAMプールの共有が重要。

## 試験頻出ポイント
- **共有可能リソース**: **Transit Gateway**・**VPCサブネット(VPC共有)**・**Route 53 Resolverルール**・**マネージドPrefix List**・**IPAMプール**・**Route 53 Resolver DNS Firewallルールグループ**。
  - NAT Gateway / IGW は**共有不可**(ひっかけ)。
- **VPC共有(サブネット共有)**: オーナーがVPC/サブネットを所有・管理、参加アカウントはサブネットに**ENI/リソースを配置**。CIDRを分散させず**IPを集約**、ピアリング不要で同一VPC内通信。
- **TGW共有**: 複数アカウントのVPCを同一TGWにアタッチする標準手段。**アタッチメント自体は各アカウント所有**、TGW所有権はオーナーが保持。
- **Organizations連携で招待不要**: 組織内共有を有効化すれば即時利用。組織外アカウントは**招待＋承諾が必要**。
- **所有権はオーナーが保持**したまま使用権だけ付与。コンシューマーは権限範囲内で操作可。
- **コンシューマー側のIAMポリシー・SCPは引き続き適用**される(共有で無効化されない)。
- **Prefix List共有**でSG/ルートテーブルが参照するCIDRを一元更新(DRY)。
- **Resolverルール共有**でハイブリッドDNSの転送設定を組織で再利用・統一。
- **RAM自体は無料**。共有したリソースの利用料は通常どおり発生。

## 比較・選択の判断
| 要件 | 解答 |
|---|---|
| 複数アカウントのVPCを同一ハブに接続 | RAMでTGW共有(ピアリングや個別TGWではない) |
| 複数アカウントを1VPCに集約しIP集約 | RAMでVPC共有(サブネット共有) |
| 組織内共有を招待なしで即時利用 | Organizations内共有を有効化 |
| SG/RTのCIDRを一元更新・配布 | マネージドPrefix ListをRAM共有 |
| ハイブリッドDNS転送ルールを組織再利用 | Route 53 ResolverルールをRAM共有 |
| 複数アカウントのENI配置先を共通化 | VPC共有(サブネット共有) |

## よく問われる上限・注意点（ひっかけ）
- **NAT Gateway / IGW はRAM共有対象外**。共有できるNWリソースの選別問題に注意。
- **リージョナルリソースは同一リージョン内のみ**共有可。**グローバルリソースのホームリージョンはus-east-1(バージニア北部)**。
- 共有してもコンシューマー側の**IAM/SCPは無効化されない**(「IAMが無効になる」は誤答)。
- **所有権は移転しない**。オーナーが保持し続け、コンシューマーは使用権のみ。
- IAMプリンシパル(ロール/ユーザー)への共有可否は**リソースタイプによって異なる**。
- 課金は共有件数・共有先数・データ転送に依存しない。RAMは無料で、リソース利用料のみ発生。
- VPC共有でアカウントごとのNAT GW/エンドポイント重複を削減でき**コスト最適化**にも寄与。
