# ルーティングと BGP（ネットワーク基礎）

> カテゴリ: ネットワーク基礎 / 重要度: ◎（最重要）
> ANS-C01 のハイブリッド接続（Direct Connect / VPN）と Transit Gateway 問題の中核。経路選択ロジックを順序立てて追えることが鍵。
> 最終更新: 2026-05-24 ／ 出典は本ドキュメント末尾

---

## 1. 概要

ルーティングは「宛先 IP をどの次ホップへ転送するか」を決める仕組みである。AWS ではルートテーブルが静的にこれを担い、Direct Connect・Site-to-Site VPN・Transit Gateway では **BGP（Border Gateway Protocol）** によって経路が動的に交換される。BGP の属性と経路選択順序、そして「最長プレフィックス一致」を理解していないと、ハイブリッド構成のトラフィックがどこを通るかを説明できない。

### なぜ ANS 試験で重要か（試験での問われ方）

- **複数経路が存在するときどれが選ばれるか**（ロンゲストマッチ → BGP 属性の順）を判断させる問題。
- **Direct Connect と VPN の優先制御**: 通常時は DX、障害時に VPN へフェイルオーバーする経路設計。
- **AS_PATH プリペンド / LOCAL_PREF** による経路の優先度操作。
- **ブラックホールルート**で特定宛先を破棄する制御。
- **経路伝播（route propagation）**: VGW / TGW がルートテーブルへ動的に経路を流し込む挙動。
- **ECMP** による帯域束ね（複数 VPN トンネルの等コスト分散）。

## 2. 中核概念

### 2.1 ロンゲストプレフィックスマッチ（最長一致）

ルーターは宛先 IP に対し、**最も具体的（プレフィックス長が最も長い）なルート**を選ぶ。これは静的/動的やプロトコルに優先する、ルックアップの大原則。

例: 宛先 `10.0.1.50` に対して次のルートがある場合、

| ルート | プレフィックス長 |
|---|---|
| 10.0.0.0/16 → TGW | 16 |
| 10.0.1.0/24 → VPC ピアリング | 24 |
| 0.0.0.0/0 → IGW | 0 |

→ 最も長い **`10.0.1.0/24`（VPC ピアリング）** が選択される。

### 2.2 ルートテーブル（AWS）

- 宛先 CIDR と **ターゲット**（IGW / NAT GW / VGW / TGW / ENI / ピアリング接続など）の対応表。
- **ローカルルート**（VPC CIDR 宛て）は最優先で削除不可。
- 明示的に関連付けられていないサブネットは**メインルートテーブル**に従う。
- 同一プレフィックス長で競合した場合の AWS 内優先順位: **ローカル > 静的（手動） > 伝播（BGP）**。さらに伝播経路同士なら DX が VPN より優先される。

### 2.3 静的ルーティング vs 動的ルーティング

| 観点 | 静的ルート | 動的ルート（BGP） |
|---|---|---|
| 設定 | 手動でプレフィックスを記述 | ネイバーと自動で経路交換 |
| 障害追従 | 自動では切り替わらない | リンク断を検知し再収束 |
| スケール | 小規模向け | 大規模・冗長構成向け |
| AWS での例 | ルートテーブルへの手動エントリ、静的 VPN | DX、動的 VPN、TGW のルート伝播 |

### 2.4 BGP の主要属性と経路選択順序（頻出）

BGP は同一プレフィックスへの複数経路から 1 本を選ぶ。AWS 文脈で重要な順序は次のとおり（上ほど優先）:

1. **最長プレフィックス一致**（より具体的な経路が常に勝つ。属性比較の前段）
2. **Weight**（Cisco 独自・ローカルルーターのみ。大きいほど優先）
3. **LOCAL_PREF（ローカルプリファレンス）**: AS 内で共有。**大きいほど優先**。自 AS からの出口（アウトバウンド）制御に使う。
4. **AS_PATH**: 経由する AS 番号の列。**短いほど優先**。`AS_PATH プリペンド`で自 AS 番号を水増しして経路を**非優先**にできる（インバウンド制御）。
5. **Origin**: IGP < EGP < Incomplete の順で優先。
6. **MED（Multi-Exit Discriminator）**: 隣接 AS への入口のヒント。**小さいほど優先**。同一隣接 AS への複数リンク間でのみ比較。
7. **eBGP > iBGP**: 外部学習経路を内部学習より優先。

**コミュニティ**: 経路に付けるタグ（例 `65000:100`）。受信側でまとめて LOCAL_PREF を変える等のポリシー適用に使う。AWS は特定のコミュニティで広報範囲（同一リージョン/同一大陸/グローバル）を制御できる。

### 2.5 eBGP と iBGP

- **eBGP**: 異なる AS 間（例: オンプレ AS とお客様側 AS と AWS）。AWS の VGW / DX / TGW とのピアは eBGP。
- **iBGP**: 同一 AS 内のルーター間。
- AWS 側のデフォルト AS 番号: VGW は **64512**、TGW は **64512** を既定とし、いずれも変更可能（DX のパブリック/プライベート VIF では自身の ASN を指定）。

### 2.6 経路伝播（Route Propagation）

VGW や TGW が BGP で学習した経路を、関連付けられた**ルートテーブルへ自動で書き込む**機能。AWS ルートテーブルで「ルート伝播を有効化」すると、オンプレから広報されたプレフィックスが手動入力なしで現れる。

### 2.7 ECMP（等コスト・マルチパス）

宛先・コストが等しい複数経路にトラフィックを分散する。

- **Site-to-Site VPN**: 1 接続あたり最大 1.25 Gbps。ECMP を有効にした **TGW** に複数トンネルを束ねると帯域を集約できる（VGW は VPN の ECMP 非対応）。
- フローは 5 タプルでハッシュされ、同一フローは同一パスを通る（順序入れ替え防止）。

### 2.8 ブラックホールルート

宛先を意図的に破棄するルート。AWS ルートテーブルでターゲットが無効化されると経路状態が **blackhole** になる（例: アタッチメント削除）。TGW のルートテーブルでは明示的にブラックホールを設定し、特定 CIDR への通信を遮断する用途にも使う。

## 3. ワークステップ例題（経路選択を手順で）

### 例題 A: 最長一致 → LOCAL_PREF → AS_PATH の順で選ぶ

宛先 `192.168.10.5` に対し次の 4 経路がある。どれが選ばれるか。

1. 経路 1: `192.168.0.0/16`, LOCAL_PREF 200, AS_PATH `[65001]`
2. 経路 2: `192.168.10.0/24`, LOCAL_PREF 100, AS_PATH `[65001 65002]`
3. 経路 3: `192.168.10.0/24`, LOCAL_PREF 200, AS_PATH `[65003 65004 65005]`
4. 経路 4: `192.168.10.0/24`, LOCAL_PREF 200, AS_PATH `[65006]`

手順:
1. **最長一致**: /24 が /16 より具体的 → 経路 1（/16）は脱落。残るは 2・3・4。
2. **LOCAL_PREF**（大きいほど優先）: 経路 2 は 100、経路 3・4 は 200 → 経路 2 脱落。残るは 3・4。
3. **AS_PATH**（短いほど優先）: 経路 3 は長さ 3、経路 4 は長さ 1 → **経路 4 が選択**。

### 例題 B: Direct Connect と VPN の優先（同一プレフィックス）

オンプレが `10.0.0.0/16` を DX と VPN の両方から AWS に広報している。両方同じプレフィックス長の場合、AWS はどちらを使うか。

手順:
1. プレフィックス長が同じ → 最長一致では決まらない。
2. AWS の伝播経路優先順位により **Direct Connect が VPN に優先**。
3. よって平常時は DX を通り、DX 障害時に VPN へフェイルオーバーする。

### 例題 C: AS_PATH プリペンドでインバウンドを誘導

2 つの DX 経由でオンプレに入りたいが、回線 B を予備にしたい。

手順:
1. オンプレから AWS への広報で、回線 B 経由の広報に**自 AS 番号を 2 回プリペンド**（AS_PATH を長くする）。
2. AWS 側から見ると回線 B の AS_PATH が長くなり**非優先**。
3. 平常時は回線 A、A 障害時に回線 B が使われる（インバウンド方向の制御は AS_PATH プリペンドが定石）。

## 5. よくある誤解・ひっかけ

- **LOCAL_PREF と MED の方向を逆に覚える**: LOCAL_PREF は**大きいほど優先（アウトバウンド制御）**、MED は**小さいほど優先（インバウンドのヒント）**。
- **属性比較が最長一致に勝つと思い込む**: どんなに LOCAL_PREF が高くても、より具体的なプレフィックスがあればそちらが先に選ばれる。**最長一致は属性比較の前段**。
- **AS_PATH プリペンドの方向**: プリペンドは AS_PATH を**長くして非優先にする**。優先したい経路にプリペンドするのは誤り。
- **VGW で VPN の ECMP ができると思う**: VGW は VPN ECMP 非対応。複数トンネルを束ねたいなら **TGW（ECMP 有効）**。
- **DX より VPN が優先されると誤解**: 同一プレフィックスなら **DX が優先**。VPN はバックアップ。
- **静的が常に BGP に勝つと誤解**: AWS では同一プレフィックス長で**静的（手動）> 伝播（BGP）**だが、まず最長一致が適用される点を忘れない。
