# IP アドレッシングとサブネッティング（ネットワーク基礎）

> カテゴリ: ネットワーク基礎 / 重要度: ◎（最重要）
> ANS-C01 のすべての設計・トラブルシューティング問題の土台。CIDR 計算は「計算できて当たり前」として出題される。
> 最終更新: 2026-05-24 ／ 出典は本ドキュメント末尾

---

## 1. 概要

IP アドレッシングとサブネッティングは、ネットワークをどのように区画化し、各区画にいくつのホストを収容できるかを決める基礎技術である。AWS では VPC・サブネット・ピアリング・Transit Gateway・Direct Connect のすべてが CIDR ブロックを前提に設計される。CIDR の計算を手早く正確に行えることが、ANS-C01 合格の前提条件になる。

### なぜ ANS 試験で重要か（試験での問われ方）

- **CIDR 表記からホスト数・サブネット数を即答**できることが求められる（`/22 は何台収容できるか` など）。
- **VLSM（可変長サブネットマスク）** で 1 つの大きな CIDR を要件に合わせて分割する設計問題。
- **アドレス重複の回避**: VPC ピアリングや TGW でつなぐ複数 VPC・オンプレの CIDR が重複してはならない、という制約。
- **AWS 予約アドレス（サブネットあたり 5 個）** を考慮した実効ホスト数の計算。
- **IP 枯渇対策**: セカンダリ CIDR、プレフィックス委任、IPv6 への移行。
- 計算ミスを誘う**ひっかけ**（予約アドレスの引き忘れ、ネットワーク/ブロードキャストの数え間違い）が頻出。

## 2. 中核概念

### 2.1 CIDR 表記とサブネットマスク

CIDR（Classless Inter-Domain Routing）表記は `ネットワークアドレス/プレフィックス長` で表す。プレフィックス長はネットワーク部のビット数（先頭から数えた `1` の数）。

```
10.0.0.0/24
└─ ネットワーク部 24 ビット ─┘└ ホスト部 8 ビット ┘
```

| プレフィックス | サブネットマスク | ホスト部ビット | 総アドレス数 (2^h) |
|---|---|---|---|
| /16 | 255.255.0.0 | 16 | 65,536 |
| /20 | 255.255.240.0 | 12 | 4,096 |
| /22 | 255.255.252.0 | 10 | 1,024 |
| /24 | 255.255.255.0 | 8 | 256 |
| /25 | 255.255.255.128 | 7 | 128 |
| /26 | 255.255.255.192 | 6 | 64 |
| /27 | 255.255.255.224 | 5 | 32 |
| /28 | 255.255.255.240 | 4 | 16 |

**覚え方**: 第4オクテットのマスク値は「256 − ブロックサイズ」。例えば /26 はブロックサイズ 64 なので 256 − 64 = 192 → `255.255.255.192`。

### 2.2 総アドレス数とホスト数の計算

- **総アドレス数** = 2^(32 − プレフィックス長)
- **一般的なネットワークの利用可能ホスト数** = 総アドレス数 − 2（ネットワークアドレスとブロードキャストアドレスを除く）
- **AWS サブネットの利用可能ホスト数** = 総アドレス数 − **5**（後述の予約 5 アドレスを除く）

### 2.3 AWS が予約する 5 アドレス（最重要・ひっかけ頻出）

AWS のサブネットでは、各サブネットの先頭 4 アドレスと末尾 1 アドレスの**計 5 アドレス**が予約されており、EC2 などに割り当てられない。`10.0.0.0/24` の例:

| アドレス | 用途 |
|---|---|
| 10.0.0.0 | ネットワークアドレス |
| 10.0.0.1 | VPC ルーター |
| 10.0.0.2 | Amazon が提供する DNS（VPC+2） |
| 10.0.0.3 | 将来利用のため AWS 予約 |
| 10.0.0.255 | ブロードキャスト（AWS では未使用だが予約） |

→ `/24`（256 アドレス）の AWS 実効ホスト数は **256 − 5 = 251**。

### 2.4 VLSM（可変長サブネットマスク）

1 つの CIDR を、必要なホスト数に応じて**異なるサイズ**のサブネットに分割する手法。大きいサブネットから順に割り当てると重複なく効率的に切れる。

### 2.5 プライベート IP 範囲（RFC 1918）

| 範囲 | プレフィックス | アドレス数 |
|---|---|---|
| 10.0.0.0 – 10.255.255.255 | 10.0.0.0/8 | 約 1,677 万 |
| 172.16.0.0 – 172.31.255.255 | 172.16.0.0/12 | 約 104 万 |
| 192.168.0.0 – 192.168.255.255 | 192.168.0.0/16 | 65,536 |

- VPC の CIDR は **/16〜/28** の範囲で指定する。
- `100.64.0.0/10`（RFC 6598 / CGNAT 範囲）も VPC で利用可能。EKS のセカンダリ CIDR などで IP 枯渇回避に使われる。

### 2.6 IPv6 基礎

- アドレス長は **128 ビット**、16 進数で 8 グループ（例 `2001:db8::1`）。
- VPC に割り当てられる IPv6 CIDR は固定の **/56**、サブネットは固定の **/64**。
- IPv6 には **NAT の概念が無い**。アウトバウンド専用には **Egress-Only Internet Gateway** を使う（インバウンドは開始させない）。
- IPv6 アドレスはすべてグローバルにルーティング可能（パブリック）。プライベート/パブリックの区別が無いため、到達制御はセキュリティグループ／NACL／ルーティングで行う。

## 3. ワークステップ例題（計算過程付き）

### 例題 A: `10.0.0.0/22` のホスト数

1. ホスト部ビット = 32 − 22 = **10 ビット**
2. 総アドレス数 = 2^10 = **1,024**
3. 一般ネットワークの利用可能ホスト数 = 1,024 − 2 = **1,022**
4. **AWS サブネット**の利用可能ホスト数 = 1,024 − 5 = **1,019**

→ 試験では「AWS のサブネット」と書かれていれば **− 5**、単なる一般ネットワークなら **− 2**。

### 例題 B: `10.0.0.0/24` を `/26` に分割する

1. /24 の総アドレス = 256、/26 の総アドレス = 64
2. 作れるサブネット数 = 256 ÷ 64 = **4 個**
3. ブロックサイズ 64 ずつ区切る:

| サブネット | 範囲 | ネットワーク | ブロードキャスト |
|---|---|---|---|
| 10.0.0.0/26 | .0 – .63 | 10.0.0.0 | 10.0.0.63 |
| 10.0.0.64/26 | .64 – .127 | 10.0.0.64 | 10.0.0.127 |
| 10.0.0.128/26 | .128 – .191 | 10.0.0.128 | 10.0.0.191 |
| 10.0.0.192/26 | .192 – .255 | 10.0.0.192 | 10.0.0.255 |

→ 各 /26 の AWS 実効ホスト数は 64 − 5 = **59**。

### 例題 C: あるIP `10.0.0.200` はどの `/26` サブネットに属するか

1. ブロックサイズ = 64
2. 200 ÷ 64 = 3 あまり 8 → 3 番目の境界 `64 × 3 = 192`
3. よって属するサブネットは **10.0.0.192/26**（範囲 .192 – .255）

### 例題 D: VLSM で `10.0.0.0/24` を要件に合わせて分割

要件: サブネット X に 100 ホスト、Y に 50 ホスト、Z に 20 ホスト。

1. X: 100 ホスト → 必要総数 ≧ 105（+5予約）→ 128 が必要 → **/25（10.0.0.0/25, .0–.127）**
2. Y: 50 ホスト → 必要総数 ≧ 55 → 64 が必要 → **/26（10.0.0.128/26, .128–.191）**
3. Z: 20 ホスト → 必要総数 ≧ 25 → 32 が必要 → **/27（10.0.0.192/27, .192–.223）**
4. 残り .224–.255 は予備。大きい順に割ると重複せず無駄が少ない。

## 4. AWS サービスとの接続

- [VPC](../../networking-content-delivery/vpc/README.md): VPC とサブネットの CIDR 設計、予約 5 アドレス、セカンダリ CIDR による拡張はこのトピックの計算が直接効く。
- [VPC ピアリング](../../networking-content-delivery/vpc/README.md): ピアリングする VPC 間で **CIDR が重複してはならない**（重複すると経路が成立しない）。
- [Transit Gateway](../../networking-content-delivery/transit-gateway/README.md): 多数の VPC・オンプレを束ねるため、組織全体の **CIDR 重複回避**と集約設計が必須。
- [Direct Connect](../../networking-content-delivery/direct-connect/README.md): オンプレと AWS のアドレス空間を重複なく設計し、BGP で広報するプレフィックスを計画する。

## 5. よくある誤解・ひっかけ

- **予約アドレスの引き忘れ**: AWS サブネットは `− 2` ではなく `− 5`。`/28`（16 アドレス）の実効ホストは 16 − 5 = **11**であって 14 ではない。
- **VPC の最小サイズ**: VPC CIDR は /16〜/28。/28 より小さい（/29 等）は指定できない。
- **プライマリ CIDR は変更不可**: 後から広げたい場合は**セカンダリ CIDR を追加**する（既存を /16→/15 のように拡張はできない）。
- **IPv6 に NAT は無い**: アウトバウンドのみ許可したいなら Egress-Only IGW。NAT Gateway を IPv6 で使おうとするのは誤り。
- **ブロックサイズの数え間違い**: /26 のブロックは 64 であって 62 ではない（62 は利用可能ホスト数）。ネットワーク/ブロードキャストを含む「総数」と「利用可能数」を混同しない。
- **172.16.0.0/12 の範囲**: 172.16.x.x〜172.31.x.x まで。172.32.x.x はプライベート範囲外。
